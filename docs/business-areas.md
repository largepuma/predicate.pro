# Predicate.pro 业务领域说明

> 版本日期：2026-04-07
> 状态：方向草案，用于统一业务叙事、产品拆解和后续验证优先级。

## 1. 总体定位

Predicate.pro 的新业务方向可以统一表述为：

**面向中文用户、创作者、开发者和高合规行业的 privacy-first AI 推理与应用平台。**

这次调整的核心不是单一产品转型，而是把公司能力聚焦到同一条主线：

- **用户价值**：更稳定地访问更少审查、更少数据留存、更贴近中文语境的 AI 能力。
- **技术底座**：模型路由、推理代理、隐私保护、日志最小化、支付与访问稳定性。
- **商业形态**：从 C 端订阅、垂直 SaaS、开发者 API 到企业级私有/可信推理。
- **差异化叙事**：不是泛泛讲“AI 工具”，而是围绕“可访问、可选择、少审查、少留存、可验证隐私”建立品牌。

四条业务线分别服务不同市场层级：

| 业务方向 | 主要用户 | 产品形态 | 变现方式 | 竞争优势 |
| --- | --- | --- | --- | --- |
| 中文市场的 Venice | 中文 AI 重度用户、内容创作者、跨境用户 | 面向个人的隐私 AI 聊天平台 | 月订阅、用量包、会员套餐 | 中文支付、访问稳定、中文模型优先、反审查叙事 |
| 垂直 Uncensored 创作/角色扮演 SaaS | 小说作者、同人创作者、角色扮演用户、桌游社群 | 垂直应用 SaaS | $9-15/月订阅、角色包、社区分销 | 长期角色记忆、双语角色库、细分创作场景 |
| Privacy API Gateway | AI 应用开发者、创业团队、安全研究者 | 隐私优先推理代理/API 网关 | API 用量计费、团队套餐、企业套餐 | 不落日志、成本归因、模型 fallback、私有模型混合接入 |
| 企业隐私合规推理 | 医疗、法律、金融、合规团队 | TEE/私有部署/可信推理解决方案 | 企业合同、PoC、年费 SLA | 可验证零数据访问、合规审计、私有化集成 |

## 2. 业务线一：中文市场的 Venice

### 2.1 定位

面向中文市场的 privacy-first、uncensored AI 聊天与推理平台。它不是简单复制 Venice，而是解决 Venice 在中文市场天然薄弱的部分：

- 国内访问不稳定。
- 缺少 Alipay 和微信支付。
- 界面、模型选择和提示词体验没有围绕中文用户优化。
- 隐私叙事与中文用户真实焦虑不完全匹配。

Predicate.pro 的切入点应聚焦在：

- **访问稳定**：通过国内 CDN、海外 Anycast、边缘代理或可替代访问链路降低连接失败率。
- **中文支付**：支持 Alipay、微信支付及适合华人用户的订阅/充值方式。
- **中文模型优先**：以 Qwen、DeepSeek 等中文语境表现更强的模型为默认入口，同时保留海外模型选择。
- **不被审查、不被存储**：宣传重点从“美国公司不会收集你”转向“你的输入不会被国内平台审查，也不会被平台长期存储”。

### 2.2 目标用户

- 希望使用更开放 AI 对话能力的中文内容创作者。
- 对国内大模型审查敏感的技术用户、研究者、开发者。
- 跨境工作、留学、海外华人等需要中文体验但不想受国内平台限制的用户。
- 希望尝试 DeepSeek/Qwen/海外模型但不想折腾 API、代理、支付和模型配置的个人用户。

### 2.3 核心功能

- 中文优先的 AI Chat 界面。
- 模型选择以 Qwen、DeepSeek 为主，附带 Claude、OpenAI、Llama、Mistral、OpenRouter 模型作为高级选项。
- 默认不保存会话，或提供本地保存/端侧加密保存选项。
- 可选会话记忆，但必须清晰标注保存边界和删除方式。
- 多线路访问策略，包括海外线路、Anycast、CDN、备用 API 路由。
- 中文支付、套餐、发票或账单记录。
- 针对中文用户的 prompt 模板，例如写作、翻译、代码、情感倾诉、法律/商业草稿、安全研究。

### 2.4 商业模式

- 免费额度：用于降低初次体验门槛。
- Pro 订阅：面向高频个人用户，按月收取固定费用。
- 用量包：面向模型重度用户，按 token 或点数消耗。
- 高级模型包：对更贵模型、长上下文、图像/语音功能单独计费。

### 2.5 主要风险

- 访问稳定性不是单点功能，而是长期运维能力，需要持续监控线路、延迟、失败率。
- 支付渠道涉及合规、风控和账户冻结风险。
- “uncensored”叙事要避免承诺非法用途，文案应聚焦表达自由、创作自由、研究用途和隐私保护。
- 如果只是换皮聊天站，容易陷入模型成本战；必须把中文体验、支付、访问和隐私叙事做成组合壁垒。

### 2.6 验证路径

1. 先做面向中文用户的最小可用聊天产品。
2. 默认接入 Qwen、DeepSeek 和 OpenRouter fallback。
3. 上线不保存会话的隐私模式和清晰的数据处理说明。
4. 通过中文 AI 社区、小红书/Telegram/Discord/独立博客进行小范围测试。
5. 用访问成功率、付费转化率、模型选择分布和留存率判断是否继续加码。

## 3. 业务线二：垂直 Uncensored 创意写作/角色扮演 SaaS

### 3.1 定位

面向创意写作、角色扮演和互动叙事场景的垂直 AI SaaS。与通用 AI 聊天平台相比，这条线的核心不是“更多模型”，而是“更懂角色、更有记忆、更贴近创作流程”。

Venice 这类通用平台可以提供 uncensored 模型能力，但在角色深度定制、长期关系感、场景工作流和社区分发上不够垂直。Predicate.pro 可以通过更具体的创作场景建立差异化。

### 3.2 目标用户

- 网文作者、同人作者、剧本作者、游戏文案。
- 角色扮演用户和虚拟陪伴用户。
- DnD、TRPG、剧本杀、跑团社群。
- 需要中英双语创作的华人创作者。

### 3.3 场景切入

优先切入更具体、付费意愿更明确的细分场景：

- **武侠小说 AI 协作者**：帮助构建门派、武功体系、人物关系、章节大纲和对白。
- **DnD 地下城主**：自动生成 NPC、遭遇战、地图描述、任务线和即时旁白。
- **剧本杀 NPC 生成器**：生成角色背景、隐藏动机、线索卡、对话风格和分支反应。
- **中英双语角色库**：服务华人创作者和海外中文创作社区，支持中英设定互转。

### 3.4 核心功能

- 角色卡创建：人物设定、语气、边界、禁忌、背景故事、关系网。
- 长期角色记忆：记住用户与角色之间的关键事件、偏好、关系变化和世界观状态。
- 世界观设定库：地点、组织、时间线、物品、能力体系、剧情伏笔。
- 场景模式：写作助手、角色扮演、剧情续写、对白打磨、世界观问答。
- 语音输入/输出：降低沉浸式对话和跑团使用门槛。
- 多语言支持：中文为核心，同时支持英文角色卡和双语输出。
- 导出能力：导出为 Markdown、角色卡、章节大纲、剧本草稿或 JSON 结构化设定。

### 3.5 商业模式

- 个人订阅：$9-15/月，面向高频创作者和角色扮演用户。
- 高级角色包：按角色库、世界观模板、类型包收费。
- 社区合作：与小说平台、桌游社群、创作者社群做分销或联名包。
- 团队/工作室套餐：面向小型游戏工作室、剧本杀工作室、内容团队。

### 3.6 差异化重点

- 不做泛聊天，而是做“角色关系和创作资产管理”。
- 记忆不是简单聊天记录，而是结构化的角色关系、剧情状态和世界观设定。
- 中文写作体验优先，尤其是武侠、仙侠、网文、剧本杀等中文语境强的类型。
- 可以复用底层 Privacy API Gateway，向用户承诺更少日志、更少审查、更高可控性。

### 3.7 主要风险

- 内容安全边界需要明确，不能把“uncensored”变成无规则内容生成。
- 角色记忆会带来数据保存与隐私承诺之间的张力，需要让用户明确选择是否保存、保存什么、如何删除。
- 角色扮演市场竞争激烈，需要通过垂直场景和中文内容库避免与大型平台正面拼规模。

### 3.8 验证路径

1. 从一个细分场景开始，例如“武侠小说 AI 协作者”或“DnD 地下城主”。
2. 做 20-50 个高质量角色/世界观模板，而不是先做大而全平台。
3. 找小型创作者社群做封闭测试，观察用户是否连续使用同一角色或世界观。
4. 用次日/7 日留存、单角色对话轮数、导出次数和订阅转化作为核心指标。

## 4. 业务线三：Privacy API Gateway

### 4.1 定位

面向开发者的 privacy-first inference proxy。它解决的不是普通用户聊天体验，而是开发者在接入多模型 AI 推理时遇到的隐私、成本、隔离、审计和 fallback 问题。

Venice 虽有公开 API，但面向企业或严肃开发者时存在空缺：

- 缺少企业级 SLA。
- 项目隔离和 cost attribution 不够强。
- 难以混合接入私有部署模型。
- 隐私承诺如果不可验证，对企业和开发者说服力不足。

Predicate.pro 可以把现有技术栈沉淀为“隐私优先 AI 网关”。

### 4.2 目标用户

- 做情感支持、心理陪伴、日记、私人助理等敏感数据应用的开发者。
- 做安全研究、红队、内容分析、开放模型测试的团队。
- 需要宽松审核但仍要合规边界的 AI 应用团队。
- 需要同时接入 OpenRouter、私有模型和多个云模型的创业公司。

### 4.3 核心功能

- OpenAI-compatible API，降低迁移成本。
- 多模型路由：OpenRouter、Qwen、DeepSeek、私有模型和其他供应商。
- Fallback 策略：按失败率、延迟、价格、模型能力自动降级或切换。
- 项目级隔离：按项目、环境、API key、团队成员区分用量。
- 成本归因：按项目、用户、模型、请求标签统计 token 和费用。
- 日志最小化：默认不保存 prompt/response，只保存必要的计费与调试元数据。
- 可验证隐私：提供日志策略、保留周期、可选审计日志、请求哈希证明或开源代理组件。
- 策略层：支持 allowlist/denylist、模型预算、速率限制、敏感场景开关。

### 4.4 商业模式

- 开发者免费层：小额度免费请求，降低试用门槛。
- 用量计费：按 token、请求量或网关加价计费。
- 团队套餐：项目隔离、成本报表、API key 管理、基础 SLA。
- 企业套餐：私有模型接入、专属线路、审计、合规文档、定制 SLA。

### 4.5 差异化重点

- 比通用模型聚合器更强调隐私、日志最小化和可验证承诺。
- 比单一模型 API 更强调 fallback、成本归因和供应商抽象。
- 比企业私有化方案更轻，适合作为创业团队从 MVP 到合规升级的中间层。
- 可以服务上层自有产品，例如中文 Venice 和创意写作 SaaS，从而让内部需求反哺开发者平台。

### 4.6 主要风险

- “不落日志”必须技术上自洽，不能影响计费、风控、滥用检测和故障排查。
- 上游模型成本和限流会直接影响毛利和稳定性。
- 开发者平台需要文档、SDK、示例和支持体系，否则转化率会低。
- 宽松审核场景容易被滥用，需要清楚定义可接受使用政策。

### 4.7 验证路径

1. 先实现 OpenAI-compatible 网关、API key、项目隔离和基础用量统计。
2. 支持 OpenRouter fallback，并在文档中说明路由策略和隐私边界。
3. 做一个公开的“不保存 prompt/response”技术说明，列出实际保存字段。
4. 找 3-5 个开发者或小团队做真实集成。
5. 用 API 日活、请求成功率、fallback 命中率、毛利率和付费团队数判断优先级。

## 5. 业务线四：企业隐私合规推理

### 5.1 定位

面向医疗、法律、金融等高合规行业的可信 AI 推理解决方案。与前三条业务线相比，这条线门槛最高、销售周期最长，但客单价和长期天花板也更高。

市场上已有 Phala、Tinfoil 等玩家用 TEE 做可验证的零数据访问 AI 推理。Predicate.pro 如果进入该方向，需要避免一开始就正面做完整 TEE 基础设施，而应从可交付的企业场景和集成能力切入。

### 5.2 目标用户

- 医疗健康应用，需要处理病历、问诊、健康记录等敏感数据。
- 法律科技应用，需要处理合同、案件材料、客户咨询和证据文件。
- 金融科技应用，需要处理交易、风控、客户资产或内部分析数据。
- 企业内部 AI 平台团队，需要在数据不出域或最小留存前提下接入模型能力。

### 5.3 产品形态

- 私有推理网关：部署在客户云环境或专有 VPC 内。
- TEE 推理 PoC：验证敏感 prompt 在可信执行环境中处理，平台无法明文访问。
- 合规审计包：数据流图、日志策略、访问控制、保留周期、供应商说明。
- 混合模型路由：企业私有模型、开源模型和外部模型按数据敏感级别分流。
- 零数据访问证明：通过远程证明、请求哈希、审计记录或开源组件增强可信度。

### 5.4 商业模式

- 付费 PoC：限定场景、限定数据范围，快速验证技术可行性。
- 年费企业合同：包含部署、维护、SLA、合规支持和安全评审。
- 专业服务：模型路由设计、隐私架构咨询、合规材料准备、私有模型集成。
- 增值模块：TEE、专属实例、私有日志系统、客户自管密钥、合规模型评测。

### 5.5 差异化重点

- 不把自己定位成单纯 TEE 基础设施供应商，而是做“企业隐私 AI 推理集成层”。
- 先用 Privacy API Gateway 打底，再逐步增加企业部署、审计和 TEE 能力。
- 面向具体行业场景产出模板，例如法律合同审查、医疗问答摘要、金融客户支持。
- 用可验证机制替代纯口头承诺，建立企业客户信任。

### 5.6 主要风险

- 企业销售周期长，需要安全评审、法务评审和定制集成，短期收入不稳定。
- TEE 技术复杂，性能、成本、供应链和可用模型范围都可能受限制。
- 医疗、法律、金融场景合规要求高，不能仅靠通用隐私文案进入市场。
- 如果没有明确的行业客户，过早投入完整平台会造成研发资源分散。

### 5.7 验证路径

1. 从 Privacy API Gateway 的企业版能力开始，而不是单独启动完整 TEE 平台。
2. 找一个高价值但边界清晰的 PoC 场景，例如法律合同摘要或金融内部知识问答。
3. 输出数据流、日志策略、部署模型、权限模型和风险边界。
4. 评估 TEE 只用于最敏感链路，避免一开始把所有推理都放进 TEE。
5. 用 PoC 付费意愿、集成周期、合规阻塞点和客户复购意向决定是否扩张。

## 6. 四条业务线之间的关系

这四个方向不应被看作彼此独立的产品，而应形成一套由浅入深的业务组合：

- **中文 Venice** 是 C 端入口，验证访问稳定、中文支付、中文模型和隐私叙事。
- **创作/角色扮演 SaaS** 是更强付费意愿的垂直 C 端/小 B 场景，验证长期记忆、角色资产和社区分发。
- **Privacy API Gateway** 是技术底座产品化，支撑内部应用，也对外服务开发者。
- **企业隐私合规推理** 是 Gateway 的高端化和合规化延伸，适合在有客户需求后逐步推进。

推荐优先级：

1. **先做 Privacy API Gateway 的基础能力**：因为它能同时支撑其他三条线，是最可复用的底座。
2. **并行验证中文 Venice 的 C 端需求**：用最小聊天产品测试支付、访问和模型偏好。
3. **选择一个创作垂直场景做深**：不要一开始做完整角色平台，先验证一个细分社区。
4. **企业合规推理先做 PoC，不重资产投入**：等 Gateway 稳定、有企业线索后再加 TEE 和私有化深度能力。

## 7. 建议的 90 天路线图

### 0-30 天：统一底座与叙事

- 明确品牌主张：privacy-first、中文友好、少审查、少留存。
- 完成 OpenAI-compatible Gateway MVP。
- 接入 Qwen、DeepSeek、OpenRouter fallback。
- 输出数据处理说明：哪些数据保存、保存多久、如何删除、哪些默认不保存。
- 做一个中文 Chat MVP，复用 Gateway。

### 31-60 天：验证付费与使用频率

- 上线中文支付或至少完成支付链路 PoC。
- 做中文 Venice 风格的聊天界面和套餐页。
- 选择一个创作垂直场景，例如武侠小说协作者或 DnD DM。
- 发布 20-50 个高质量角色/模板。
- 找首批种子用户测试留存和付费意愿。

### 61-90 天：开发者化与高端线索

- 完善 API key、项目隔离、用量统计、成本归因。
- 发布开发者文档、示例代码和隐私白皮书草案。
- 对外招募 3-5 个开发者团队试用 Gateway。
- 梳理企业版能力清单：SLA、私有模型接入、VPC 部署、审计日志。
- 与潜在法律/金融/医疗场景客户沟通 PoC，但不急于重投入 TEE。

## 8. 近期需要决策的问题
